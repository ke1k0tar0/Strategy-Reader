# Product Requirements Document (PRD)

## Project Name

Trading Strategy Optimization Recommendation Engine

---

# 1. Overview

## Objective

Build a web-based system that reads trading strategy experiment data from Google Sheets, analyzes historical parameter changes and their outcomes, and recommends the currently optimal parameter set for a strategy under specific market conditions.

The system is intended to function as a lightweight quant-research assistant that continuously evaluates historical strategy tuning experiments and identifies parameter combinations associated with higher profitability and better execution quality.

This project is designed for rapid AI-assisted development ("vibe-coding"), so the architecture and implementation should prioritize:

* simplicity
* modularity
* readability
* deterministic logic
* AI-friendly code structure

The initial version should avoid unnecessary complexity such as distributed systems, deep ML pipelines, or database infrastructure.

---

# 2. Core Problem

Trading strategies are manually tuned over time by adjusting parameters and observing outcomes.

Currently:

* parameter experiments are stored in Google Sheets
* results are manually reviewed
* no automated optimization exists
* difficult to identify the best parameter combinations
* difficult to understand which changes improved performance

The system should automate this analysis process.

---

# 3. Product Goals

## Primary Goals

1. Read strategy experiment data from Google Sheets
2. Parse parameter snapshots from JSON
3. Analyze historical parameter performance
4. Detect high-performing parameter combinations
5. Recommend the best current parameter configuration
6. Filter recommendations by strategy and market condition
7. Produce deterministic optimization output

---

# 4. Non-Goals

The MVP should NOT include:

* brokerage integrations
* live trading execution
* automated order placement
* deep learning
* reinforcement learning
* distributed systems
* database infrastructure
* user authentication
* multi-user support
* real-time websocket systems
* advanced backtesting engine

---

# 5. Data Source

## Google Sheets

The system uses Google Sheets as the only data source.

Data is manually maintained by strategy researchers.

The backend reads sheet data periodically or on-demand.

---

# 6. Google Sheet Structure

Each row represents one strategy experiment or optimization attempt.

## Columns

| Column                           | Description                        |
| -------------------------------- | ---------------------------------- |
| Date                             | Date of experiment                 |
| Scope                            | Strategy identifier                |
| Parameter Set (JSON snapshot)    | Strategy parameters in JSON format |
| Hypothesis                       | Reason for parameter change        |
| Change                           | Description of modifications       |
| Stop conditions (capital safety) | Risk management constraints        |
| Success metric                   | Expected optimization target       |
| Duration                         | Trading session duration           |
| Market Conditions (vol regime)   | Market regime classification       |
| Fills                            | Fill percentage                    |
| PnL                              | Profit and loss result             |
| Top 3 Gate Reasons               | Main execution blockers/reasons    |
| Verdict                          | Final experiment evaluation        |
| Notes                            | Additional comments                |

# 7. Functional Requirements

# 7.1 Google Sheets Reader

## Requirements

* Connect to Google Sheets API
* Read rows from configured sheet
* Convert rows into structured objects
* Handle missing or malformed rows gracefully

## Expected Behavior

The system should:

* fetch all rows
* normalize column names
* return structured strategy experiment records

---

# 7.2 Parameter JSON Parser

## Requirements

The system must:

* parse the "Parameter Set (JSON snapshot)" column
* flatten parameters into machine-readable fields
* support nested JSON structures if needed

## Example Input

```json
{
  "min_ccs": 0.55,
  "max_ccs": 0.67,
  "entry_cap": 242
}
```

## Example Output

```json
{
  "min_ccs": 0.55,
  "max_ccs": 0.67,
  "entry_cap": 242
}
```

## Validation Rules

* invalid JSON should not crash the system
* malformed rows should be skipped with warnings
* parameter keys should remain consistent

---

# 7.3 Strategy Filtering

## Requirements

The system must support filtering by:

* strategy (`Scope`)
* market condition
* date range
* verdict
* minimum PnL
* minimum fills

---

# 7.4 Optimization Engine

## MVP Optimization Logic

The initial version should use deterministic statistical analysis only.

No AI model training is required initially.

## Required Logic

The engine should:

* identify profitable parameter combinations
* rank experiments by weighted score
* detect recurring successful parameter ranges
* calculate average performance per parameter set
* filter out poor-performing configurations

---

# 7.5 Scoring Engine

## Purpose

Convert experiment outcomes into a comparable optimization score.

## Example Formula

```text
score =
(PnL * 0.5)
+ (Fills * 0.3)
- (RiskPenalty * 0.2)
```

## Notes

Scoring formula should:

* be configurable
* support future weighting changes
* remain deterministic

---

# 7.6 Recommendation Engine

## Requirements

The system should generate:

* recommended parameter set
* confidence score
* expected PnL
* expected fill rate
* supporting historical evidence

## Example Output

```json
{
  "strategy": "1H Horizon",
  "marketCondition": "Bearish",
  "recommendedParameters": {
    "min_ccs": 0.60,
    "max_ccs": 0.65
  },
  "confidence": 0.78,
  "expectedPnL": 3.1,
  "expectedFillRate": 87
}
```

---

# 7.7 Explanation Engine

## Purpose

Explain WHY a parameter set is recommended.

## Example

```text
This parameter range historically produced higher fill rates and reduced entry rejection under bearish market conditions.
```

This can initially be template-based and deterministic.

No LLM integration required for MVP.
